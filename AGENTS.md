# DatumDock 协作与开发规则

## 产品目标

DatumDock 是一款本地优先的视觉数据集管理与标注桌面软件。用户先将图片及其标注统一纳入数据集池，再按配置将数据可重复地划分并导出为可直接训练的 YOLO 数据集目录。标注是数据集治理流程中的核心能力之一，而不是产品的唯一中心。

产品采用“内部资料库 → 受管数据集 → 数据集池”的层级。内部资料库由软件自动维护，不作为用户可见的工作区；用户启动后在存档式主页创建或打开数据集，不选择项目根目录。每个数据集都是独立存档，拥有隔离的图片池、标注、标签、模型配置、索引、复核状态和回收站。

每个数据集维护一套独立标签集。标签集既服务于训练，也服务于人的理解：标签必须支持机器可用的英文名称，以及面向标注人员的中文别名和说明。跨数据集复制、移动、模板或合并操作必须显式比较标签映射，不能依赖隐藏的项目父级推断兼容性。

新建数据集可选择已有数据集作为模板并复制其配置。数据集复制、移动、合并或标签集合并前，必须比较标签集签名；只有稳定标签 ID、类别 ID、英文训练名和状态映射完全兼容时才允许自动合并，任何差异都必须先展示给用户并阻止不安全操作。

每个数据集还可维护独立的自动标注模型库。模型文件受管存放在该数据集模型目录；导入、更新和删除模型只影响该数据集的模型资源，不得影响其他数据集池或既有人工标注。

## 必读文件与优先级

每次开始工作时，按以下顺序阅读：

1. `AGENTS.md`：本文件，定义协作和工程规则。
2. `docs/DATASET_LIBRARY.md`：最新且优先的存档式主页、内部资料库和旧结构迁移决定。
3. `docs/PRD.md`：产品范围与用户需求。
4. `docs/ROADMAP.md`：当前任务顺序与状态。
5. `docs/ACCEPTANCE.md`：每阶段的可验证完成条件。
6. `docs/ARCHITECTURE.md` 与 `docs/UX.md`：实现边界和交互约定。
7. `docs/VISUAL_DESIGN.md`：现代视觉 v2、组件语言、参考边界和截图验收。
8. `docs/X_ANYLABELING_BASELINE.md`：与 X-AnyLabeling 对标的能力等级、质量门槛与追踪规则。
9. `docs/X_ANYLABELING_INTEROP.md`：X-AnyLabeling/LabelMe 数据集导入、导出和无损保留边界。
10. `docs/CODE_STYLE.md`：Python 格式化、静态检查、中文注释和测试规则。

冲突时，以用户在当前对话中的明确指令为最高优先级；其次是验收标准、产品需求、架构和路线图。

## 建议技术栈

- Python 3.11+
- PySide6：桌面界面与画布交互
- OpenCV：图像读取、缩放与处理
- Pydantic：配置和标注数据模型
- pytest：自动化测试
- LabelMe JSON：内部编辑和兼容 X-AnyLabeling 的标注格式
- X-AnyLabeling/LabelMe 目录：外部数据集互操作格式
- YOLO Detection：MVP 的训练集导出格式
- PyInstaller：将应用及 Python 运行时打包为 Windows 可执行文件
- Inno Setup：生成 Windows 安装包、开始菜单项与卸载程序

在没有明确需求前，不要添加云服务、登录、数据库、遥测或联网依赖。

## 工作方式

- 从 `docs/ROADMAP.md` 中优先级最高且未完成的项目开始。
- 每次仅实现一个可独立验证的子功能；完成后运行相关检查并更新 Roadmap。
- 保持模块小而清晰；业务逻辑不要直接耦合到窗口控件。
- 产品必须使用原生桌面 GUI；核心功能不得依赖终端操作才能完成。
- 标注工作台的目标布局固定为：顶部数据集切换与导入/导出主操作栏、左侧选择/矩形框/AI 等画布工具、中央图片画布、右侧当前图片标注和带状态的虚拟图片列表。实现不得重新引入左侧工作区或项目树。
- 标注工作台左上角在空间允许时必须清楚显示完整 `DD + DatumDock` 字标，并与首页完整 Logo 保持同一视觉尺寸。常规窗口下以 170 逻辑像素可见宽度为基准、允许范围为 160～180 逻辑像素，可见图形高度不低于 24 逻辑像素；不得只增大带透明留白的图标画布。Logo 是不可点击的静态品牌展示，其左侧必须提供带 tooltip 和无障碍名称的独立返回按钮；空间不足时字标才收敛为 `DD` 标记。
- 只要鼠标指针能够映射到实际图片像素，画布的水平和竖直十字辅助线就必须持续跟随指针；该行为与选择、矩形创建、框移动/缩放或中键平移等当前操作无关。默认选择模式、没有选中任何矩形、一次性画框完成并自动退回选择模式后也必须显示。只有指针离开实际图片、空画布或图片加载失败时才隐藏，不能只在画框期间显示。
- 选择/编辑模式必须改变实际系统鼠标指针的图标来提示矩形标注的可用操作，而不是改变标注框样式：框内可移动区域为四向移动指针，左右/上下控制点分别为水平/垂直缩放指针，四个角点为对应方向的对角缩放指针。矩形创建和中键平移的鼠标指针具有更高优先级；只读或不可编辑 shape 不得显示虚假的移动/缩放反馈。
- 标注画布的图片后方使用冷调浅色底板，不再使用大面积深色背景。矩形工具允许在中央底板上开始或结束拖拽/两点创建；底板坐标必须先钳制到最近的图片边或角再形成预览和标注，最终保存坐标始终位于图片范围内。选择工具点击底板不得意外创建标注，侧栏和工具栏永远不参与吸附。
- 画布鼠标滚轮规则固定为：普通滚轮竖直滚动，`Alt + 滚轮` 水平滚动，`Ctrl + 滚轮` 以指针下图片位置为锚点放大/缩小。`Ctrl` 与 `Alt` 同时按下时缩放优先；缩放范围仍为 `1%～6400%`，且只改变视图，不进入快捷键注册、标注历史、自动保存或复核状态。
- 图片复核状态以整张图片为单位，用户可见值只允许待复核和已完成。新导入图片使用空复核值且不显示第三种状态；无标注是筛选条件，损坏/未知标签/恢复失败是独立健康诊断。零框图片可明确确认为已完成，并以零框事实表达负样本。
- 数据集主页必须提供可折叠五步快速开始和学习中心。DatumDock 全流程、YOLO Detection、数据划分/泄露和导出训练准备等核心教程以中英文离线资源随安装包提供；教程不得遮挡数据集主入口、绕过业务确认、加载广告或把“数据集导出”误写成应用内训练。
- 视觉实现必须遵守 `docs/VISUAL_DESIGN.md`：主页借鉴 Scratch 的友好与清晰引导，标注页借鉴 X-AnyLabeling 的画布优先与紧凑效率，但只使用 DatumDock 自有品牌和图标。默认 Qt 灰色、暖灰/灰绿旧式管理软件外观、立体控件、密集分隔线和像素级复制第三方界面均为不合格。
- 所有新增或修改的代码注释、docstring、TODO/FIXME 和日志信息默认使用中文；标识符、依赖名称和行业标准字段保持英文。注释必须解释原因、约束、风险或数据安全边界，不得机械复述代码。用户可见系统界面必须支持简体中文与英文，并默认使用中文。
- Python 代码必须遵循 `pyproject.toml` 的 Ruff 规范：提交前运行 `python -m ruff check src tests`、`python -m ruff format --check src tests` 和相关 pytest；禁止无中文原因的 `# noqa`、`# type: ignore` 或跳过测试。完整规则见 `docs/CODE_STYLE.md`。
- 所有 Markdown 文档以中文为主，并在文末保留对应的英文摘要；新增 Markdown 必须遵守同一语言规范。
- 数据集池中每个样本必须有稳定身份；导出划分必须支持固定随机种子，以便重复得到相同结果。
- 单个数据集至少应支持一万张常见静态图片的受管、检索、标注与导出。列表、缩略图、标签检查、导入、重命名、删除和导出必须使用索引、分页/虚拟化和后台任务，禁止一次性把全部图片或缩略图载入内存。
- 导入到数据集的数据必须复制到该数据集的受管数据池；不同数据集的样本、标注和标签映射不得混用，导出时只能使用当前数据集的样本。外部源文件始终保持不动。
- MVP 标注形状限定为矩形框，标注框不增加难度、遮挡、备注等额外属性。其他 shape 和属性必须在用户明确需要且格式、模型和导出闭环设计完成后才加入。
- 数据集池内图片可按数据集命名规则批量重命名。重命名必须保持稳定样本 ID 不变，并同步更新配套标注文件、索引、缩略图和 `imagePath` 等文件名引用。
- 从标注页面删除样本时，必须在确认后删除该样本的图片、标注文件、索引记录、缩略图和临时自动标注等派生信息；不得留下孤立文件或无图片的标注记录。导入前的外部源文件不在删除范围内。
- 删除受管样本时，用户可选择移入软件回收站或永久删除；“少量”的回收站阈值在全局设置中配置，对大批量删除可直接永久删除。两种方式均必须在确认对话框中明确目标数量和影响范围，并在相关设置项旁提供悬停帮助说明。
- 主页必须允许永久删除整个受管数据集。归档是可恢复选择，首版不另建整数据集回收站；永久删除前必须展示图片、标注、标签、模型、索引、缩略图、样本回收站和缓存等影响统计，要求输入数据集名称并进行二次确认。删除只能作用于该数据集 UUID 对应的内部受管目录，不得触碰外部来源或既有导出/备份目录；未解决的标注保存失败或运行中的数据集任务必须先处理。删除使用恢复清单和同卷暂存目录，崩溃或清理失败不得留下仍登记但无法打开的半删除数据集，也不得伪报成功。
- 标签的稳定 ID、训练类别 ID 和英文名称一经用于标注或导出，不应在未提供迁移方案时任意改动；中文别名和说明可以安全地完善。
- 标注 UI 必须优先帮助人快速理解标签：支持按英文名、中文别名、同义词和描述搜索，并在选择列表中显示中文信息。
- 鼠标悬停在可交互标签或画布矩形标注上时必须有克制、即时的视觉反馈，体验可参考 X-AnyLabeling，但只能使用 DatumDock 自有设计令牌。快速标签卡片、标签下拉候选、右侧当前标注项和标签管理表格行至少使用浅蓝背景、细边框或轻微阴影表达 hover；画布矩形使用略高的填充/边框强度反馈。已选中标签必须比 hover 更明确，并额外使用勾选、控制柄、左侧指示条或加粗边框，不能只依靠颜色。悬停、移出和键盘焦点不得修改标签、最近使用标签、标注历史、自动保存或复核状态。
- 同一数据集的每个活动标签必须拥有唯一且可辨识的显示颜色。颜色由数据集标签集持久化管理，在标签表、画布 shape、筛选、标签检查和自动标注待确认结果中保持一致；用户可手动调整但不能与其他活动标签重复。
- 数据集必须提供独立的标签管理页面。修改英文训练名、类别 ID 或其他会影响标注含义的字段前，必须预检受影响样本并经用户确认后批量迁移；迁移失败时不得留下标签集与标注文件不一致的状态。
- 标签管理页面必须能按当前数据集标签提取对应样本的虚拟检查集合。检查集合不复制或移动任何文件，并可回到原样本位置。
- 从工作台返回主页或打开其他数据集时，必须先处理当前保存失败的待处理标注；切换后 UI、筛选、类别和临时导出上下文都应切换到目标数据集。
- 同一图片的后台加载结果必须按请求代号和文档版本双重校验；人工编辑后迟到的较低版本结果不得覆盖较新的内存标注或保存检查点。只有用户明确确认“从磁盘重新加载”时，才允许丢弃当前内存版本。
- 异步切图必须分别维护导航目标、加载中的样本和已经加载完成的画布样本。切图开始后旧画布立即只读；只有 `current_image_id`、已加载 sample ID、画布图片 ID 与 `AnnotationDocument.sample_id` 完全一致时才允许编辑或构造保存请求。后台标注读取不得把同一数据集尚未完成的活动保存清单当成崩溃恢复现场。
- 导出不得修改原始数据集池；所有导出均写入用户当次指定的新目录。
- 导出模型类型、划分比例、随机种子和输出路径仅用于当次导出，不在数据集中保存导出方案或导出记录。
- 导出功能应按模型格式实现为可扩展的导出器；MVP 首先支持 YOLO Detection，后续可增加其他模型或格式而不改动数据集池和标注核心。
- 导入 ONNX、PT 等模型时，先自动探测格式、文件完整性、输入输出张量、尺寸、类别名称和内嵌元数据，再生成模型运行配置；探测结论必须可见且可由用户复核或修正。
- `.pt` 并非单一通用格式。仅对已支持的 PyTorch/TorchScript/Ultralytics 变体自动配置；无法安全识别的 PT 文件必须明确报出不支持原因，不得猜测模型结构后运行。
- 自动标注首发适配 Ultralytics YOLO 检测模型和兼容的自训练 YOLO 模型；模型二进制不包含在数据集备份包内，需由用户单独传递和导入。
- 自动标注结果应直接写入当前数据集的标注工作流，并带“待复核”、模型来源和置信度；不得覆盖人工标注。待复核图片第一次发生有效人工编辑并保存成功后自动变为已完成；无需修改时使用 `review.mark_completed` 明确确认。
- 自动标注推理优先使用经验证可用的 GPU 后端，否则回退 CPU。首次 CPU 推理必须解释 GPU 未启用原因，允许继续或打开内置 GPU 配置指引。
- 导入图片时对完全相同内容进行检测。发现重复时必须展示已有图片与待导入图片的对比对话框，允许用户取消或确认同时保留两份样本。
- 对近似图片建立相似组，不自动删除或合并样本。导出时同一已确认相似组必须整体分配到同一个 train/val/test 集合；划分器在保持组完整的前提下尽量接近用户设定比例。
- X-AnyLabeling 互操作导入时，矩形 shape 转为 DatumDock 可编辑标注；当前不支持的 shape、flags、attributes、group_id、description、difficult 和 score 等字段必须保留为兼容负载并在 X-AnyLabeling 导出时原样还原，禁止静默丢失。
- X-AnyLabeling/LabelMe 的 `rectangle` 兼容传统两个对角点以及能够证明轴对齐且包含完整四角的四点表示；二者必须由预检、正式提交、SQLite 框数和反向标签索引共用的分类函数判断。旋转、退化、重复角点或无法证明轴对齐的四点内容只读保留并明确警告，不得冒充可编辑框或静默丢失。
- 一个完整用户标注操作只能产生一个撤销历史节点、一个文档版本和一个自动保存请求。双击改派标签、模态窗口关闭后的鼠标释放或没有几何变化的拖动不得制造冗余保存；保存队列必须在处理下一项前同步登记上一成功版本和摘要。
- 标签集在工作台打开期间发生新增、编辑、归档、恢复或导入修订时，工作台必须从同一次 Gateway 快照同步刷新标签内容和标签集修订号；禁止只刷新候选标签却继续用旧修订号保存。真正的并发修订必须以独立诊断阻止覆盖，不能误报为磁盘权限。
- 快速标签窗口关闭后的标签快照刷新只能更新标签候选、颜色、说明和修订号，不得重建图片查询模型、改变分页或把当前图片重置为第一张。标签改派必须同时校验弹窗打开时的样本 ID、shape ID、文档版本与关闭后的已加载样本；不满足时明确拒绝，不能把第二张图片的修改应用到第一张或静默丢弃。
- 标签相关操作采用立即持久化，不依赖切图、离开页面、关闭窗口或手动按 `Ctrl+S` 才保存。标签管理中的新增、编辑、归档、恢复和安全显示字段修改，在用户提交一个合法变更后立即原子写入 `label-set.json`、递增修订号并回读验证；矩形标注的新增、移动、缩放、删除、撤销、重做和标签替换，在一次有效操作结束时立即提交不可变 LabelMe/SQLite 自动保存快照。连续拖动只在释放时提交一次，尚未通过字段校验的输入和已取消的对话框不属于有效变更。
- 用户可见的“待复核”只允许由模型自动标注新增或改变标注时写入。任何有效人工标注修改都必须把标注内容与图片级 `completed` 状态放在同一个恢复型保存事务中立即提交；保存失败时两者均不得单边成功或显示为已保存，内存修改必须保留并提供准确诊断和重试。纯人工新建标注从首次成功保存起即为已完成，人工修改待复核结果后也立即成为已完成。
- 标注保存问题的临时诊断日志只能写入应用资料库根目录的有界 `logs/`，不得写入数据集目录、外部来源或预览模式；日志不得包含图片字节、`imageData` 或完整标注负载，也不得成为保存成功、恢复或数据事实的前提。实机根因确认后必须把临时默认开关恢复为关闭。
- 导出 X-AnyLabeling/LabelMe 交换目录时始终导出范围内的 PNG；只有图片至少包含一个可编辑矩形或只读兼容 shape 时才生成同名 JSON。完全没有 shape 的图片不得生成空 JSON，但只含兼容 shape 的图片仍须生成 JSON 以保证无损往返。
- 在功能稳定后，维护可重复的打包脚本；Windows 首发交付物为可直接安装和卸载的安装包，而非要求用户安装 Python。
- 所有系统 UI 文案必须通过本地化资源提供，禁止在窗口和控件中散落硬编码的中英文字符串。语言切换只能改变应用界面，不能改写数据集标签、描述、文件名、标注或模型元数据。
- 所有用户可配置操作快捷键必须集中注册到 `ActionRegistry`，页面不得创建未登记的应用快捷键。设置页面自动枚举注册表，支持修改、清空、冲突替换和单项/分组/全部恢复；修改后即时生效并原子持久化，失败时运行时与文件配置一起回滚。
- 任何新依赖都要记录在 `README.md` 或项目依赖文件中。
- 不覆盖用户已有改动；发现冲突时先说明影响。
- 遇到影响产品范围、数据格式或总体架构的选择时，说明备选方案并等待确认。

## English Summary

DatumDock is a native desktop GUI with a game-save-like managed dataset home and a canvas-focused annotation workspace. Valid label-definition changes are saved atomically as soon as they are submitted, while rectangle creation, editing, deletion, undo/redo, and label reassignment enqueue one immutable autosave snapshot immediately after each effective operation. Pending review is reserved for model-generated changes; every successful manual annotation edit commits the image as completed in the same recoverable transaction. Model inference, exports, backups, and packaging remain future work.
