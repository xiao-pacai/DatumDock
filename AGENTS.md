# DatumDock 协作与开发规则

## 产品目标

DatumDock 是一款本地优先的视觉数据集管理与标注桌面软件。用户先将图片及其标注统一纳入数据集池，再按配置将数据可重复地划分并导出为可直接训练的 YOLO 数据集目录。标注是数据集治理流程中的核心能力之一，而不是产品的唯一中心。

产品采用“工作区 → 项目 → 数据集 → 数据集池”的层级：一个工作区可登记多个项目，每个项目可包含多个独立数据集；每个数据集拥有隔离的数据集池，并引用项目级标签集。

每个项目维护一个项目级标签集，供该项目的所有数据集共用。标签集既服务于训练，也服务于人的理解：标签必须支持机器可用的英文名称，以及面向标注人员的中文别名和说明。

新建数据集可选择已有数据集作为模板并复制其配置。数据集复制、移动、合并或标签集合并前，必须比较标签集签名；只有稳定标签 ID、类别 ID、英文训练名和状态映射完全兼容时才允许自动合并，任何差异都必须先展示给用户并阻止不安全操作。

每个项目还可维护独立的自动标注模型库。模型文件受管存放在项目模型目录；导入、更新和删除模型只影响该项目的模型资源，不得影响其他项目、数据集池或既有人工标注。

## 必读文件与优先级

每次开始工作时，按以下顺序阅读：

1. `AGENTS.md`：本文件，定义协作和工程规则。
2. `docs/PRD.md`：产品范围与用户需求。
3. `docs/ROADMAP.md`：当前任务顺序与状态。
4. `docs/ACCEPTANCE.md`：每阶段的可验证完成条件。
5. `docs/ARCHITECTURE.md` 与 `docs/UX.md`：实现边界和交互约定。
6. `docs/X_ANYLABELING_BASELINE.md`：与 X-AnyLabeling 对标的能力等级、质量门槛与追踪规则。
7. `docs/X_ANYLABELING_INTEROP.md`：X-AnyLabeling/LabelMe 项目导入、导出和无损保留边界。
8. `docs/CODE_STYLE.md`：Python 格式化、静态检查、中文注释和测试规则。

冲突时，以用户在当前对话中的明确指令为最高优先级；其次是验收标准、产品需求、架构和路线图。

## 建议技术栈

- Python 3.11+
- PySide6：桌面界面与画布交互
- OpenCV：图像读取、缩放与处理
- Pydantic：配置和标注数据模型
- pytest：自动化测试
- LabelMe JSON：内部编辑和兼容 X-AnyLabeling 的标注格式
- X-AnyLabeling/LabelMe 目录：外部项目互操作格式
- YOLO Detection：MVP 的训练集导出格式
- PyInstaller：将应用及 Python 运行时打包为 Windows 可执行文件
- Inno Setup：生成 Windows 安装包、开始菜单项与卸载程序

在没有明确需求前，不要添加云服务、登录、数据库、遥测或联网依赖。

## 工作方式

- 从 `docs/ROADMAP.md` 中优先级最高且未完成的项目开始。
- 每次仅实现一个可独立验证的子功能；完成后运行相关检查并更新 Roadmap。
- 保持模块小而清晰；业务逻辑不要直接耦合到窗口控件。
- 产品必须使用原生桌面 GUI；核心功能不得依赖终端操作才能完成。
- 所有新增或修改的代码注释、docstring、TODO/FIXME 和日志信息默认使用中文；标识符、依赖名称和行业标准字段保持英文。注释必须解释原因、约束、风险或数据安全边界，不得机械复述代码。用户可见系统界面必须支持简体中文与英文，并默认使用中文。
- Python 代码必须遵循 `pyproject.toml` 的 Ruff 规范：提交前运行 `python -m ruff check src tests`、`python -m ruff format --check src tests` 和相关 pytest；禁止无中文原因的 `# noqa`、`# type: ignore` 或跳过测试。完整规则见 `docs/CODE_STYLE.md`。
- 所有 Markdown 文档以中文为主，并在文末保留对应的英文摘要；新增 Markdown 必须遵守同一语言规范。
- 数据集池中每个样本必须有稳定身份；导出划分必须支持固定随机种子，以便重复得到相同结果。
- 单个项目至少应支持一万张常见静态图片的受管、检索、标注与导出。列表、缩略图、标签检查、导入、重命名、删除和导出必须使用索引、分页/虚拟化和后台任务，禁止一次性把全部图片或缩略图载入内存。
- 导入到数据集的数据必须复制到该数据集的受管数据池；不同数据集的样本、标注和标签映射不得混用，导出时只能使用当前数据集的样本。外部源文件始终保持不动。
- MVP 标注形状限定为矩形框，标注框不增加难度、遮挡、备注等额外属性。其他 shape 和属性必须在用户明确需要且格式、模型和导出闭环设计完成后才加入。
- 数据集池内图片可按数据集命名规则批量重命名。重命名必须保持稳定样本 ID 不变，并同步更新配套标注文件、索引、缩略图和 `imagePath` 等文件名引用。
- 从标注页面删除样本时，必须在确认后删除该样本的图片、标注文件、索引记录、缩略图和临时自动标注等派生信息；不得留下孤立文件或无图片的标注记录。导入前的外部源文件不在删除范围内。
- 删除受管样本时，用户可选择移入软件回收站或永久删除；“少量”的回收站阈值在全局设置中配置，对大批量删除可直接永久删除。两种方式均必须在确认对话框中明确目标数量和影响范围，并在相关设置项旁提供悬停帮助说明。
- 标签的稳定 ID、训练类别 ID 和英文名称一经用于标注或导出，不应在未提供迁移方案时任意改动；中文别名和说明可以安全地完善。
- 标注 UI 必须优先帮助人快速理解标签：支持按英文名、中文别名、同义词和描述搜索，并在选择列表中显示中文信息。
- 同一项目的每个活动标签必须拥有唯一且可辨识的显示颜色。颜色由项目标签集持久化管理，在标签表、画布 shape、筛选、标签检查和自动标注待确认结果中保持一致；用户可手动调整但不能与其他活动标签重复。
- 项目必须提供独立的标签管理页面。修改英文训练名、类别 ID 或其他会影响标注含义的字段前，必须预检受影响样本并经用户确认后批量迁移；迁移失败时不得留下标签集与标注文件不一致的状态。
- 标签管理页面必须能按项目标签提取对应样本的虚拟检查集合。检查集合跨该项目的所有数据集查询包含目标标签的图片，不复制或移动任何文件，并可回到原数据集与样本位置。
- 快速切换项目或数据集时，必须先处理当前未保存标注；切换后 UI、筛选、类别和导出配置都应切换到目标数据集上下文。
- 导出不得修改原始数据集池；所有导出均写入用户当次指定的新目录。
- 导出模型类型、划分比例、随机种子和输出路径仅用于当次导出，不在项目中保存导出方案或导出记录。
- 导出功能应按模型格式实现为可扩展的导出器；MVP 首先支持 YOLO Detection，后续可增加其他模型或格式而不改动数据集池和标注核心。
- 导入 ONNX、PT 等模型时，先自动探测格式、文件完整性、输入输出张量、尺寸、类别名称和内嵌元数据，再生成模型运行配置；探测结论必须可见且可由用户复核或修正。
- `.pt` 并非单一通用格式。仅对已支持的 PyTorch/TorchScript/Ultralytics 变体自动配置；无法安全识别的 PT 文件必须明确报出不支持原因，不得猜测模型结构后运行。
- 自动标注首发适配 Ultralytics YOLO 检测模型和兼容的自训练 YOLO 模型；模型二进制不包含在项目备份包内，需由用户单独传递和导入。
- 自动标注结果应直接写入当前项目/数据集的标注工作流，并带“待人工复核”、模型来源和置信度；不得覆盖人工标注。复核状态以整张图片为单位，人工可修改漏标、误标、标签和框位置后将样本标记为已复核或有问题。
- “有问题”是图片级待处理状态。用户完成复核并解决问题后，必须将状态切换为“已复核”并移除问题标记；问题状态不得与已复核同时存在。
- 自动标注推理优先使用经验证可用的 GPU 后端，否则回退 CPU。首次 CPU 推理必须解释 GPU 未启用原因，允许继续或打开内置 GPU 配置指引。
- 导入图片时对完全相同内容进行检测。发现重复时必须展示已有图片与待导入图片的对比对话框，允许用户取消或确认同时保留两份样本。
- 对近似图片建立相似组，不自动删除或合并样本。导出时同一已确认相似组必须整体分配到同一个 train/val/test 集合；划分器在保持组完整的前提下尽量接近用户设定比例。
- X-AnyLabeling 互操作导入时，矩形 shape 转为 DatumDock 可编辑标注；当前不支持的 shape、flags、attributes、group_id、description、difficult 和 score 等字段必须保留为兼容负载并在 X-AnyLabeling 导出时原样还原，禁止静默丢失。
- 在功能稳定后，维护可重复的打包脚本；Windows 首发交付物为可直接安装和卸载的安装包，而非要求用户安装 Python。
- 所有系统 UI 文案必须通过本地化资源提供，禁止在窗口和控件中散落硬编码的中英文字符串。语言切换只能改变应用界面，不能改写项目标签、描述、文件名、标注或模型元数据。
- 所有用户可配置操作快捷键必须集中注册。设置页面允许用户修改绑定；修改后即时生效、持久化为全局应用偏好，并在冲突或保留组合时阻止保存并说明原因。
- 任何新依赖都要记录在 `README.md` 或项目依赖文件中。
- 不覆盖用户已有改动；发现冲突时先说明影响。
- 遇到影响产品范围、数据格式或总体架构的选择时，说明备选方案并等待确认。

## 完成定义

仅当当前阶段在 `docs/ACCEPTANCE.md` 中的所有条目已满足、应用可以启动、相关测试通过，才把该阶段标记为完成。对外宣称“核心完成”或“可替代 X-AnyLabeling 核心工作流”前，还必须满足 `docs/X_ANYLABELING_BASELINE.md` 中对应等级的全部必需项；不得以界面相似或单一功能可用代替完整工作流验证。

## English Summary

DatumDock is a native desktop GUI application. Use PySide6 for the interface, PyInstaller for executable bundling, and Inno Setup for a Windows installer. Code comments and logs are Chinese by default. System UI supports Simplified Chinese and English, defaults to Chinese, and changes language without mutating project content. Every Markdown document is Chinese-first and ends with an English summary.
