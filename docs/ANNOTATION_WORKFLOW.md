# DatumDock 标签与矩形标注工作流

> 状态：2026-07-20 步骤四及 A0.5～A0.8 已实现并复验。A0.8 最终按用户反馈让工作台与首页使用同一 170px 基准完整字标，保证默认选择模式持续辅助线，并接入 `Ctrl + 滚轮` 指针锚定缩放。模型推理、YOLO 导出、完整 X-AnyLabeling 目录导入导出与备份不属于本步骤。

## 1. 用户工作流

1. 从主页打开一个受管数据集，在“标签管理”中预先建立标签。
2. 每个标签包含稳定 UUID、类别 ID、英文训练名、中文别名、描述、同义词、唯一颜色和活动/归档状态。
3. 回到标注工作台，从右侧图片列表选择图片，在左侧选择矩形工具后画框。
4. 画布和右侧标注列表双向选择；矩形可移动、使用八个控制柄缩放、删除或改派标签。
5. 新建、移动、缩放、删除、换标签、撤销、重做和复核状态变化都会立即进入串行自动保存队列。
6. 纯人工标注保存成功后默认为已完成；模型写入接口把图片标为待复核，第一次有效人工编辑保存成功后自动完成，也可直接“确认已完成”。
7. 在标签图片检查页选择一个标签，可分页查看所有包含该标签的图片，并双击跳回对应图片和首个目标框。

没有活动标签时，矩形工具保持禁用并提示先管理标签。归档标签不能用于新框，但历史框仍可显示、移动、删除或改派。

### 1.1 画布交互整改（A0.4～A0.7 已实施）

以下状态机已经接入普通模式和独立预览模式，并由 pytest-qt 覆盖：

1. 用户按 `R` 进入一次性矩形创建状态。
2. 用户可按下左键拖拽并释放完成矩形，也可以第一次单击确定一个角点、第二次单击确定对角点完成矩形；两种方式已经正式确认。
3. 第一次单击后显示稳定锚点、动态矩形轮廓、实时宽高和十字辅助线。第二点可在任意方向，提交前统一换算为规范的左上/右下原图像素坐标。
4. 两次单击不等同于双击：矩形创建模式中的两个独立点击负责起点和终点；双击已有矩形仍只打开快速更改标签小窗。
5. 第二点与第一点形成零面积时不提交，锚点继续保留以便重新点击；`Esc` 清除锚点并无副作用退出。
6. 创建成功后生成一个历史节点、排入一次自动保存，并自动回到选择/编辑状态；不得继续连续画框。
7. `Esc`、切换工具或切换图片会取消未完成的草稿；取消本身不保存、不改变复核状态。
8. 只要指针处于实际图片范围内，水平和竖直辅助线就在所有工具与操作状态下持续跟随，并裁切于图片边界；选择/编辑、矩形创建、框移动/缩放和中键平移都不能令其消失。只有离开实际图片、空画布或加载失败时隐藏。该规则取代此前“中键平移时隐藏”和“只在画框上下文显示”的行为。
9. 标注采用标签色外框和同色低透明填充，外框的不透明度高于填充；选中框继续显示八点控制柄和独立高亮。
10. 按住鼠标中键拖拽平移图片；滚轮竖直滚动；`Alt + 滚轮` 水平滚动。三种操作不得误触发画框、移动标注或改变当前工具。
11. 缩放由工具栏按钮、比例入口、`100%`、适配窗口和 `Ctrl + 滚轮` 提供。`Ctrl + 滚轮` 向上放大、向下缩小；指针位于图片内时以该像素为锚点，否则以可视区域中心为锚点；`Ctrl + Alt + 滚轮` 仍按缩放处理。
12. 缩放范围至少到 `6400%`，并提供 `100%` 与适配窗口恢复入口。高倍率只修改视图变换，不创建巨型图片副本。
13. 在任意倍率下，矩形、控制柄、命中检测和十字线都从同一套视图/原图坐标转换得出；保存到 LabelMe 的值始终是原图像素坐标。
14. 高倍率时框线、控制柄与标签浮层保持稳定屏幕尺寸，用户可用中键及两种滚轮映射查看图片边缘和角落。
15. 实际系统鼠标指针图标随操作语义变化，但不改变矩形标注外观：选择/编辑模式命中框内可移动区域时使用四向移动指针；左右、上下和四角控制点分别使用水平、垂直及对应对角缩放指针。按下拖动期间保持操作指针，释放或取消后重新按悬停命中计算。
16. 鼠标指针样式优先级固定为中键平移、矩形创建、缩放控制点、框内移动、普通选择；只读兼容 shape 和不可编辑文档不得显示移动/缩放反馈。指针图标变化不属于标注编辑，不进入历史、自动保存或复核状态机。
17. 图片后方使用冷调浅色底板。矩形工具收到的拖拽起点/终点或两次单击点即使位于中央底板，也先经同一视图坐标转换，再逐轴钳制到最近图片边或角；预览、宽高和提交全部使用钳制结果。
18. 图片内点保持真实像素位置；图片左/右外侧分别钳制到 `x=0 / image_width`，上/下外侧分别钳制到 `y=0 / image_height`，角外区域同时钳制两个轴。反向绘制后仍规范化为左上/右下坐标。
19. 钳制后零面积继续遵守既有零面积规则，不创建、保存或改变复核状态。最终 `RectangleShape` 和 LabelMe JSON 永远不含越界坐标，Service 仍独立校验并拒绝绕过 UI 的越界输入。
20. 底板吸附只属于可编辑图片的矩形创建状态；选择工具点击底板不画框，中键平移优先，中央画布外的界面区域不参与。A0.5 辅助线仍裁切在图片内，底板上只显示必要的边缘吸附反馈。

实现时必须保持所有框坐标仍为原图像素坐标，视图滚动和平移不得写入 LabelMe JSON，也不得改变图片级复核状态。

### 1.2 A0.8 用户实机整改（已完成）

1. 标注工作台左上角在 1366×768 及以上常规窗口中显示清楚可辨的完整 `DD + DatumDock` 字标；与首页统一使用裁去透明安全区后的 170 逻辑像素基准，可见宽度允许 160～180 逻辑像素、可见图形高度不低于 24 逻辑像素。Logo 不响应点击，其左侧的独立返回按钮负责回到主页；窄窗口切换到 `DD` 标记。
2. 辅助线必须在默认选择工具、没有选中任何矩形、没有矩形草稿和一次性画框完成后持续工作；只要鼠标位于实际图片内，两条线就跟随鼠标。普通模式与 `--ui-preview` 必须共享同一规则。
3. `Ctrl + 滚轮` 改为固定缩放手势，优先级高于 `Alt + 滚轮` 横向滚动；缩放保持 1%～6400% 范围，并尽量固定指针下原图像素的屏幕位置。
4. 上述三项都只影响品牌展示或视图交互，不得修改标注文档、撤销历史、自动保存、复核状态或数据集内容。
5. A0.8 已增加真实普通模式鼠标移动与滚轮事件回归，并重新生成中文/英文 × 三分辨率 × 100%/125%/150% DPI 证据；旧 A0.5 截图仅保留为历史。

#### 图片切换默认按键

- `A` 是上一张图片的出厂默认按键，`D` 是下一张图片的出厂默认按键；两者都由统一 `ActionRegistry` 提供，可在设置中改绑或清空。
- “恢复全部默认按键”后，这两个操作恢复为 `A / D`。菜单和 tooltip 同步显示当前有效绑定，不把字母硬编码在工作台。
- 文本输入焦点优先于图片导航；在搜索、标签名称或描述中输入 `a`、`d` 不切图。
- 快捷切图与鼠标选择图片执行相同的自动保存离开保护、跨页定位和数据集隔离规则。保存未完成时等待，保存失败时阻止切换并提供处理选项。

### 1.2 已实施的快速更改标签小窗

1. 双击画布中的矩形框，打开可调整大小的小窗并预选当前标签；打开小窗不立即改变文档。
2. 顶部搜索框始终可见，支持中文别名、英文训练名、同义词和描述；旁边提供“新建标签”。
3. 中部使用响应式标签卡片：宽度增加时自动增加列数，宽度减少时自动换行；卡片过多时仅该区域纵向滚动。
4. 每张卡片显示标签颜色、中文别名和英文训练名；描述按空间显示为次行或 tooltip。当前项与候选项具有非颜色的选中反馈。
5. 底部“确认 / 取消”固定可见。确认后标签改派形成一个历史节点并排入一次自动保存；取消、关闭或 `Esc` 不改变矩形。
6. 双击命中后必须结束该次鼠标移动事务并吞掉弹窗关闭后的配对释放事件；改派与鼠标释放不得分别生成两个版本。移动/缩放是否变化只比较矩形几何坐标。
7. 人工编辑会立即使编辑前启动的同图后台加载结果失效；即使迟到结果使用当前请求代号，较低文档版本也不得覆盖较新的内存标注。用户在诊断中明确选择“从磁盘重新加载”时，才清空内存版本并通过受管 Gateway 重新读取。
6. 快速新建使用真实 `LabelSetService` 和完整标签校验，不创建仅存在于当前标注中的临时类型。创建成功后新标签成为准备选择项。
7. 小窗至少支持 `Ctrl+F` 聚焦搜索、方向键选择、`Enter` 确认和 `Esc` 取消；鼠标和键盘操作结果一致。

小窗尺寸作为全局应用偏好跨会话记忆，并在当前屏幕可用区域内钳制。快速新建标签是独立的数据集事务：创建成功后即永久保留；随后取消外层改派只取消矩形改派，不删除新标签。这两个边界均有回归测试。

快速新建、标签管理或导入使标签集修订号变化时，工作台必须从同一次 Gateway 快照同时更新标签候选和修订号，再提交外层改派。只刷新候选列表而沿用旧修订号会被并发保护拒绝；该情况使用独立的“标签集修订冲突”诊断，不得误报为权限问题。

### 1.2.1 最近使用标签跟随（已完成）

1. 每个数据集分别维护当前标注会话的“最近使用标签”，不得在数据集之间共享或串用。
2. 用户使用 `test1` 成功创建矩形后，新建矩形默认使用 `test1`；随后成功使用或确认 `test2` 创建矩形后，下一次新建矩形必须默认使用 `test2`，不能回退到首次标签或标签表第一项。
3. 双击已有矩形并确认改派标签，也视为一次实际使用；改派成功后，后续新框默认使用改派后的标签。
4. 仅打开标签小窗、搜索或悬停候选、选择后取消、关闭窗口、按 `Esc`、创建零面积框或保存失败，都不能改变最近使用标签。
5. 最近标签被归档、删除或在当前数据集中不存在时立即失效；下一次画框应要求用户选择活动标签，不得静默映射到其他标签。
6. 最近使用标签至少在当前应用会话内跨图片保持；本轮不要求关闭应用后继续记忆，后续若持久化必须存入当前数据集显示配置并保持数据集隔离。
7. 该状态只决定下一次创建矩形的默认标签，不改写已有矩形、不产生独立历史节点，也不单独触发自动保存或复核状态变化。

### 1.3 已实施的右侧列表删除

1. 单击右侧当前标注项，会选中同一个 shape UUID 对应的画布矩形。
2. 右侧列表或画布具有操作焦点时，按 `Delete` 调用同一条“删除当前标注”命令。
3. 删除对象是当前图片中的矩形标注实例，不是数据集标签定义；其他图片中的同类框保持不变。
4. 一次删除只形成一个历史节点和一个不可变自动保存快照；画布、列表、SQLite 反向索引和框数必须一起更新，`Ctrl+Z` 可恢复。
5. 如果待复核图片删除了最后一个框，这次有效人工编辑保存成功后自动完成，并以零框表示无目标；保存失败则仍为待复核。
6. 搜索、标签创建或其他文本输入控件具有焦点时，`Delete` 仅编辑文字，不得冒泡为标注删除。

### 1.4 标签相关变更即时保存（已完成）

1. 数据集标签定义的新增、编辑、归档、恢复，以及别名、描述、同义词和颜色修改，在合法表单提交后立即原子保存 `label-set.json`、递增标签集修订号并回读验证。
2. 矩形新增、移动、缩放、删除、替换标签以及实际改变文档的撤销/重做，在一次有效操作结束时立即排入串行自动保存；用户无需切图、返回主页、关闭软件或另按保存键。
3. 连续拖动只在鼠标释放时形成一个历史节点、一个文档版本和一个保存请求；搜索、选择、取消标签窗口、零面积或没有实际变化的操作不保存。
4. 快速新建标签与矩形改派是两个事务：新标签创建成功后立即持久化并保留，确认改派后再立即保存标注文档；取消外层改派不删除已保存的新标签。
5. `pending_review` 只由模型自动标注写入。纯人工创建的标注首次保存即为 `completed`；对待复核标注进行任何有效人工修改时，标注内容和 `completed` 在同一次恢复型保存中提交。
6. UI 在请求入队后显示“正在保存”，只有文件和 SQLite 均提交并回读验证后显示“已保存”。失败时保留内存修改、选中状态和最新快照，提供准确详情与重试，不能让标注或复核状态单边成功。

### 1.5 标签悬停反馈（已完成）

1. 画布矩形、快速标签卡片、下拉候选、右侧当前标注项和标签管理表格行在鼠标进入时显示轻微填充增强、浅蓝高亮、细边框或轻阴影，离开后恢复。
2. hover 只说明“当前指针可作用于此标签”；已选中状态使用控制柄、更明确的背景、勾选或左侧指示条，不能被较弱 hover 覆盖。
3. 键盘焦点具有等价的焦点反馈；归档、只读和禁用标签不显示虚假的可编辑或可点击反馈。
4. 悬停、移出或仅移动键盘焦点不改变 shape 的 `label_id`，不更新最近使用标签，也不产生历史、文档版本、保存请求或复核状态变化。
5. 颜色、边框、阴影和过渡来自集中式设计令牌，并尊重系统减少动态效果；视觉可参考 X-AnyLabeling 的操作清晰度，但不复制其资产。

## 2. 标签规则

- 标签和标签集使用规范 UUID；类别 ID 在包括归档标签在内的整个标签集中唯一且不复用。
- 活动英文训练名按大小写不敏感规则唯一；活动颜色精确唯一。
- 中文别名必填；描述和同义词可选。标签搜索覆盖训练名、别名、描述和同义词。
- 修改别名、描述、同义词或颜色只写 `label-set.json`，不会触碰任何标注 JSON。
- 上述标签定义变更在合法提交后立即保存并递增修订号；页面离开和应用关闭不是正常保存触发器。
- 修改类别 ID 需要影响预览和修订号确认，但不会改写 LabelMe JSON。
- 修改英文训练名会备份并原子迁移受影响 JSON 的标准 `shape.label`，同时保留稳定 `datumdock_label_id`。
- 训练名迁移中断时，`recovery/label-migrations/` 中的有限操作记录用于启动恢复；恢复不会扫描一万份无关 JSON。

## 3. LabelMe 存储事实边界

标注文件固定存放在：

```text
datasets/<dataset_uuid>/pool/annotations/<image_stem>.json
```

JSON 是标注交换事实来源，SQLite v3 是分页查询与状态摘要事实来源。DatumDock 的受管 JSON：

- 使用标准 `shape_type: "rectangle"`、英文训练名和两个原图像素对角点；
- 保存私有稳定 `datumdock_shape_id` 与 `datumdock_label_id`，避免显示字段变化破坏内部引用；
- 新文件使用 `imageData: null`；已有合法 `imageData`、根字段和未知字段保持不变；
- 按原顺序保存矩形与未支持 shape，不会把兼容 shape 统一移动到文件前后；
- 保留 `flags`、`attributes`、`score`、`group_id`、`description`、`difficult` 等兼容负载；
- 未来执行外部目录导出时会递归剔除全部 `datumdock_` 私有字段。

步骤五已在受管池边界外增加安全目录配对、显式标签映射、`labels.txt` 和独立原子导出。对外 JSON 会递归剔除全部 `datumdock_*` 私有字段；混合 shape 顺序、未知 shape 和兼容字段已通过自动化双向回归。第三方 X-AnyLabeling 实际打开、编辑和回导仍是未通过的硬性验收。

## 4. SQLite v3

`DatasetSampleRepository` 在事务中逐级升级到 v3，新增并维护：

- `annotation_count`、`annotation_state`、`annotation_version`、`annotation_sha256` 和更新时间；
- `sample_labels(sample_id, label_id, shape_id)` 反向索引；
- 标签使用量、标签筛选、复核状态筛选和跨页样本定位查询；
- 可空的 `review_status`，只允许 `pending_review` 与 `completed`；
- v2 五状态及遗留 `auto_pending_review / reviewed` 到双状态的诊断式迁移。

升级只迁移结构和已有索引行，不同步解析全部 JSON。标签检查、图片列表和万级定位均查询 SQLite，每页最多 200 条，不为一万张图片创建一万个控件。

## 5. 原子保存与恢复

一次真实保存依次完成：

1. 验证数据集、样本、标签、图片尺寸、矩形坐标、磁盘摘要和受管路径。
2. 在 `recovery/annotation-operations/<dataset_uuid>/<operation_uuid>/` 写入清单、原 JSON 备份和候选 JSON。
3. 写入并验证临时 JSON，刷新后原子替换正式文件。
4. 重读正式 JSON 并计算 SHA-256。
5. 在一个 SQLite 事务内提交摘要、版本、框数、反向标签索引和复核状态。
6. 删除恢复标记并报告该文档版本已保存。

若 JSON 已提交但 SQLite 失败，服务立即原子恢复旧 JSON，SQLite 仍保留旧摘要和旧复核状态，候选文件留作重试。只有旧 JSON 也无法恢复时，样本才进入 `recovery_required` 并禁止继续编辑。损坏 JSON 保持原字节并以只读异常显示。

每个自动保存请求持有不可变文档快照；同一数据集队列按 sample UUID 分别维护最后版本和摘要，不能把一张图的并发前提用于另一张图。切图、切数据集、返回主页或关闭窗口时，正在保存会先等待；失败时必须选择“重试保存 / 放弃内存修改 / 取消”。

有效人工标注命令提交后立即进入该队列，并将目标复核状态设为 `completed`；来源为模型的实际标注变化才把状态设为 `pending_review`。状态与标注摘要、版本、框数和反向标签索引在同一个 SQLite 事务中提交，保存失败时不得提前改变持久化复核状态。

保存线程在开始下一个排队请求前同步登记上一版本的已提交版本和磁盘摘要。失败详情保留请求 ID、编辑类型、样本/shape、基准/目标/当前版本、预期/实际摘要、标签修订、异常链和恢复状态；只有真实权限异常才显示权限原因。状态栏的失败文本可点击打开详情，并按失败可重试性提供“重试最新版本”或“从磁盘重新加载”。

### 5.1 临时自动保存调试日志

用户实机日志已经定位“第三张及后续图片提示标注尚未安全保存”的根因：导航目标在异步加载开始时已经切换，但画布仍短暂保留上一张图片的可编辑文档，标签改派因此把新 sample ID 与旧文档版本、摘要和框组合为一个请求。保存服务正确拒绝了该跨样本请求，原 JSON 未被覆盖。

后续实机日志又定位到独立的界面状态问题：在第二张图片确认标签改派后，快速标签窗口会刷新标签修订号；旧实现复用了整页 `retranslate_ui()`，其中状态筛选器重建时发出查询信号，样本模型被重置并默认选中第一张。安全身份校验因此拒绝了已经不属于当前画布的第二张改派，所以日志中没有第二张保存请求。这不是磁盘权限或 JSON 写入失败。

当前修复规则如下：

- 工作台分别维护导航目标、加载中的 sample ID 与已经加载完成的 sample ID；三者不再由一个字段隐式代表。
- 切图开始即清空旧标注文档、撤销历史和选择，并将画布设为只读；只有目标图片字节与标注文档被当前请求一起接受后才恢复编辑。
- 任何编辑和保存入队前都验证 `current_image_id == loaded_sample_id == document.sample_id == canvas.image.id`。不一致时只记录诊断，不创建历史、版本或保存请求。
- `AnnotationSaveRequest.sample_id` 取自不可变文档，不再取自可能已提前变化的导航目标。
- 同一数据集存在在途自动保存时，后台标注预览和画布读取先等待短事务结束，再执行恢复检查，不能把尚未清理的活动恢复清单当成崩溃现场。
- 快速标签窗口关闭后只局部更新标签控件、画布标签快照和修订号，不重建图片模型或筛选条件；改派前再次校验弹窗打开时的样本、shape、文档版本和当前已加载样本。
- 同一图片的重复在途选择会合并；旧加载结果和非当前样本的保存完成回调不能覆盖当前画布摘要、复核状态或文档。

普通模式继续暂时启用结构化 JSON Lines 日志，直到用户用原数据集完成相同切图与改派流程的实机复验。

- 默认路径为 `%LOCALAPPDATA%\DatumDock\logs\annotation-debug.log`；测试或开发覆盖 `DATUMDOCK_DATA_DIR` 时，日志位于覆盖根目录的 `logs` 子目录。
- 日志记录图片请求代号、数据集/样本 UUID、文档与 SQLite 版本、摘要前缀、标签集修订、保存请求 ID、JSON 发布、SQLite 提交、回滚与 UI 接收结果，不记录图片字节或 `imageData`。
- 单文件上限为 2 MiB，最多保留当前文件和 3 个轮转备份。日志写入失败不能改变业务保存结果。
- `--ui-preview` 不创建日志；没有实际加载或保存标注的普通空数据集也不应仅因启动、切换语言或关闭窗口而创建日志。
- 临时关闭方法：在同一个 PowerShell 窗口执行 `$env:DATUMDOCK_ANNOTATION_DEBUG = "0"` 后再启动应用。删除该环境变量或设为 `1` 可恢复记录。
- 复现时依次编辑第一、第二、第三张及后续图片；提示出现后关闭应用，并保留 `annotation-debug.log` 与存在时的 `annotation-debug.log.1`。定位根因并取得实机复验后，应把源码默认值恢复为关闭。

## 6. 步骤四历史复核状态

> 本节表格仅保留 schema v2 的历史行为证据；当前运行版本已经由 6.1 节的 schema v3 双状态模型取代。

| 状态 | 规则 |
| --- | --- |
| `unreviewed` | 新导入图片的默认状态。 |
| `pending_review` | 第一次人工画框后进入；已完成图片发生任何框编辑后也回到此状态。 |
| `completed` | 用户明确确认，且至少有一个有效矩形。 |
| `completed_negative` | 用户明确确认无目标，且矩形数量必须为零。 |
| `issue` | 用户明确标记有问题；编辑后仍保持，直到重新确认完成。 |

缺图、损坏 JSON、未知标签或待恢复操作属于派生异常，不允许用户手工设置。删除最后一个框后进入待审核，不会自动变成负样本。仅存在矩形也不会自动变成已完成。

### 6.1 已实施的双状态复核闭环

| 用户可见状态 | 内部值 | 新进入规则 |
| --- | --- | --- |
| 待复核 | `pending_review` | 模型自动标注新增或改变当前图片的标注。 |
| 已完成 | `completed` | 纯人工创建标注默认进入；待复核模型结果发生一次有效人工编辑并保存成功，或经人工无修改确认。 |

- 新导入且没有标注的图片使用空复核值，不显示第三种状态徽标；可通过“无标注”条件筛选。
- 健康异常继续独立显示，不属于复核状态。
- 人工标注图片默认已完成。模型待复核图片发生创建、移动、缩放、删除、换标签或实际改变文档的撤销/重做后，在该标注保存成功时自动完成。
- 选择、查看详情、打开后取消标签小窗、缩放/平移画布、搜索和筛选不属于有效人工编辑，不能改变待复核状态。
- 人工编辑与 `completed` 必须在同一次自动保存快照和 SQLite 事务中提交；失败时两者都保持待处理，持久化状态仍为待复核。
- `review.mark_completed` 用于人工检查后无需修改的图片，只改变图片状态、不修改矩形；成功后原子写入 `completed`，失败时仍显示待复核。
- 零框图片可以人工确认成已完成并作为负样本，不再使用 `completed_negative` 的独立徽标。
- 不再提供未复核、待审核、有问题或已完成（无目标）等额外用户状态；旧值按架构文档的 v2→v3 规则安全迁移。
- “确认已完成”注册为 `review.mark_completed`，出厂默认键为 `S`，并允许用户改绑、清空或恢复默认。

矩形必须使用有限数值、面积大于零并完全位于图片边界内。A0.7 画布会在 UI 预览前把底板输入钳制到边界，但服务层仍不能信任 UI；面积大于零的极小框允许保存但产生质量警告，零面积、非有限或真正越界框会阻止保存。

## 7. 与图片治理协同

- 批量重命名同步修改标注文件名、`imagePath`、SQLite 路径、摘要和更新时间；提交失败恢复原文件字节及原摘要。
- 移入回收站时保留样本 UUID、标注文件、缩略图和反向索引；恢复名称冲突时按当前命名规则安全改名并重新计算 JSON 摘要。
- 永久删除同时删除受管图片、标注、缩略图、样本行和标签反向索引；外部来源文件永不属于删除目标。
- 缩略图“标注预览”开启时，仅对视图请求的当前页可见项按需加载矩形，关闭后清除叠加缓存，不全量扫描数据集。

## English Summary

Valid label-definition changes persist immediately, and effective rectangle edits enqueue one autosave snapshot. Interactive label surfaces provide subtle hover feedback, while selected and keyboard-focused states remain stronger and accessible; hover has no data or autosave side effects. Pending review remains model-only, and successful manual edits commit completed status in the same recoverable transaction.
